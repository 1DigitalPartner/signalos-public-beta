# SignalOS Design Partner Program

The SignalOS Design Partner Program is for selected companies, agencies, consultants, and operating teams that want to evaluate SignalOS against real commercial workflows.

This program goes beyond general beta testing.

Design partners help validate whether SignalOS can reduce analysis time, improve recommendation quality, create safer execution workflows, and produce measurable business value.

---

## Who the program is for

The program is designed for:

- B2B companies;
- digital agencies;
- growth consultancies;
- technical marketing teams;
- revenue operations teams;
- product-led growth teams;
- founders managing active acquisition and conversion systems;
- operators responsible for analytics, content, campaigns, or implementation handoff.

The strongest candidates have a real workflow, a defined business problem, and enough operating context to evaluate outcomes.

---

## Suitable use cases

Examples include:

- improving a commercial website;
- finding missing conversion paths;
- detecting weak audience capture;
- improving campaign-to-pipeline routing;
- auditing analytics and tracking coverage;
- standardizing client audits;
- reducing manual recommendation writing;
- improving handoff from marketing to development;
- turning approved findings into controlled GitHub pull requests;
- monitoring whether implemented fixes improve the asset over time.

A design partner does not need to test every feature.

The program should focus on one or two meaningful workflows.

---

## What design partners receive

Selected participants may receive:

- controlled access to the hosted SignalOS beta;
- a larger evaluation allowance than general beta testers;
- a guided onboarding session;
- a defined evaluation mission;
- priority consideration for relevant integrations;
- direct product feedback sessions;
- early access to selected capabilities;
- support in structuring the test environment;
- consideration for preferential early-access pricing;
- optional recognition as a founding design partner.

Access level and support depend on fit, capacity, technical readiness, and the proposed use case.

---

## What SignalOS expects

Design partners should be prepared to:

- define a real commercial problem;
- document the current process;
- identify a baseline;
- complete an agreed evaluation period;
- use only authorized assets and repositories;
- review all generated work;
- provide structured product feedback;
- provide honest commercial feedback;
- identify adoption blockers;
- discuss measurable or observable outcomes;
- respect security and confidentiality boundaries.

The program is not intended for passive access without testing.

---

## Recommended evaluation period

A typical evaluation lasts:

- one week for a focused workflow;
- two weeks for repeated scans and recommendation review;
- up to four weeks for a broader agency or team workflow.

The scope should remain narrow enough to produce clear evidence.

---

## Evaluation framework

### 1. Define the problem

Document:

- the asset or workflow;
- the current business problem;
- who is affected;
- why the problem matters;
- the current tools;
- the current manual process;
- the expected outcome.

### 2. Establish a baseline

Capture:

- time normally required;
- number of people involved;
- quality or consistency problems;
- implementation delay;
- missing measurement;
- current conversion or pipeline indicators;
- current review and approval process.

### 3. Run SignalOS

Use SignalOS to:

- onboard the asset;
- run diagnostics;
- review recommendations;
- select relevant fixes;
- generate implementation packages;
- test controlled execution where appropriate;
- verify completed work.

### 4. Record the result

Capture:

- findings considered useful;
- findings considered weak;
- approved actions;
- work generated;
- external execution results;
- time saved;
- workflow improvement;
- observed business impact;
- missing capabilities;
- trust or governance concerns.

### 5. Review commercial value

Discuss:

- whether the workflow should be repeated;
- who would use the product;
- who would approve payment;
- expected monthly value;
- preferred pricing model;
- adoption blockers;
- renewal requirements;
- required integrations;
- required security or compliance controls.

---

## Success criteria

A successful design partnership should produce at least three of the following:

- a validated commercial use case;
- measurable time savings;
- improved recommendation consistency;
- faster implementation handoff;
- a safer execution workflow;
- improved tracking or conversion coverage;
- clearer prioritization;
- a reusable operating workflow;
- a product requirement supported by real evidence;
- a credible willingness-to-pay signal.

A successful evaluation does not require every SignalOS recommendation to be accepted.

Negative evidence is useful when it is specific and grounded.

---

## Agency evaluation track

Agencies and consultancies may evaluate SignalOS for:

- client asset onboarding;
- repeatable audit workflows;
- standardized recommendation quality;
- implementation package generation;
- developer handoff;
- controlled repository changes;
- client-ready reporting;
- multi-client governance requirements.

Important questions include:

- Can the workflow be repeated across clients?
- Does it reduce analyst preparation time?
- Are findings specific enough for client delivery?
- Can recommendations be reviewed before client presentation?
- Are client workspaces and access separation required?
- What export and branding capabilities are necessary?
- What pricing model fits agency economics?

---

## B2B team evaluation track

B2B teams may evaluate SignalOS for:

- owned-audience growth;
- website conversion;
- content-to-pipeline routing;
- campaign measurement;
- analytics coverage;
- recommendation prioritization;
- marketing and development coordination;
- controlled execution.

Important questions include:

- Does SignalOS identify problems the team had missed?
- Are findings connected to revenue or pipeline?
- Which team owns the workflow?
- Does the approval model fit existing governance?
- Which connectors are required?
- Can outcomes be measured after implementation?
- Would the product become part of a weekly operating process?

---

## Technical operator evaluation track

Technical operators may focus on:

- OAuth authorization;
- connector token handling;
- exact target restrictions;
- approval invalidation;
- application-level approval;
- isolated execution branches;
- commit and pull request generation;
- verification;
- rollback;
- audit metadata.

The objective is to determine whether AI-assisted execution can remain controlled, reviewable, and operationally credible.

---

## Data and security boundaries

Design partners must not provide:

- passwords;
- private keys;
- access tokens;
- session cookies;
- service credentials;
- regulated personal data;
- confidential customer datasets;
- production secrets;
- unauthorized repositories;
- assets they do not have permission to evaluate.

GitHub tests should use:

- a dedicated repository;
- a safe documentation file;
- no production deployment configuration;
- no customer secrets;
- no automatic merge.

Security-sensitive findings must follow `SECURITY.md`.

---

## Confidentiality

Public beta feedback may be discussed through public GitHub channels only when it contains no confidential information.

Commercial, customer-specific, architectural, or security-sensitive feedback should use an agreed private channel.

TanziTech will not publish a participant's:

- company name;
- logo;
- testimonial;
- case study;
- performance data;
- internal workflow;
- identity;

without permission.
